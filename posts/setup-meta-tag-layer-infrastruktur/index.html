Setup Meta Tag Noindex/Nofollow di Layer Infrastruktur dengan Traefik

Solusi untuk mencegah mesin pencari mengindeks website internal menggunakan Traefik reverse proxy tanpa perlu memodifikasi kode aplikasi.

Umumnya tools internal dijaga dan disembunyikan dari hasil pencarian mesin pencari seperti Google atau Bing dengan alasan skor SEO (dan keamanan tentunya). Caranya adalah dengan memberi tahu mesin pencari agar tidak mengindeks halaman pada suatu website dan juga tidak meng-crawl website tersebut.

Untuk memberitahu mesin pencari agar tidak mengindeks halaman pada suatu website, diperlukan penambahan meta tag noindex dan nofollow. Sementara untuk menjaga dari crawler mesin pencari, kita perlu memasang robots.txt yang berisi rules bot mesin pencari dan path apa saja yang boleh ditelusuri oleh si bot. Kedua proses ini membutuhkan perubahan kode sehingga biasanya task-nya akan di-assign ke tim developer.

Hal yang gampang, cukup menambahkan sebaris tag HTML di sela-sela <head>...</head>.

Namun apa jadinya jika kita tidak punya kuasa atas kode dari aplikasi yang kita gunakan?. Misalnya saja, dalam kasus ini kita men-deploy sebuah internal tool dari proyek open source menggunakan pre-build image resminya. Men-fork kemudian memodifikasi sebaris kode untuk membuat docker image sendiri rasanya malah tidak efektif dan efisien.

Untungnya, dalam software engineering, selalu ada jalan lain menuju Roma. Dalam kasus kali ini kita memiliki akses dan memiliki kontrol penuh terhadap infrastruktur. Dimana kita menggunakan Dokploy, sebuah platform pengelolaan infrastruktur berbasis Docker Compose dengan Traefik sebagai load balncer dan reverse proxy bawaannya.

Anggaplah kita men-deploy Chatwoot, sebuah open source apps untuk customer engagement dengan struktur docker compose sebagai berikut:

services:
  app-rails:
    image: chatwoot/chatwoot:v4.0.3
    entrypoint: docker/entrypoints/rails.sh
    command: [... && rails s -p 3000 -b 0.0.0.0']
    ...
  pgvector:
    image: pgvector/pgvector:pg14
    ...
  other-dependencies:
    ...
networks:
  ...
volumes:
  ...

Setup noindex dan nofollow pada Reverse Proxy dengan Header X-Robots-Tag#

Ternyata ada cara lain untuk menambahkan aturan noindex dan nofollow tanpa perlu mengubah kode aplikasi, yaitu dengan mendefinisikannya pada header X-Robots-Tag di HTTP response dari server. Header ini memiliki fungsi yang sama dengan meta tag yang umum digunakan dan didefinisikan secara ekplisit pada halaman web.

Quote

The X-Robots-Tag can be used as an element of the HTTP header response for a given URL. Any rule that can be used in a robots meta tag can also be specified as an X-Robots-Tag.

â€” Google Search Central1

Dengan setup yang kita miliki (Dokploy), kita dapat menambahkan header ini dengan tagging/labels middleware Traefik yang didefinisikan pada docker-compose.

Caranya dengan menambahkan middleware untuk meng-inject header pada response website. Middleware ini dibuat dengan direktif Traefik sebagai label pada docker compose, sehingga service app-rails menjadi seperti:

services:
  app-rails:
    image: image-docker-yang-ada
    labels:
      - "traefik.http.routers.chatwoot.middlewares=noindex-headers@docker"
      - "traefik.http.middlewares.noindex-headers.headers.customresponseheaders.X-Robots-Tag=noindex, nofollow"
  ...

traefik.http.routers.chatwoot.middlewares=noindex-headers@docker, memasang middleware ke setiap router/path aplikasi web, nama middleware-nya noindex-headers.

traefik.http.middlewares.noindex-headers.headers.customresponseheaders.X-Robots-Tag=noindex, nofollow, definisi middleware-nya. Memerintahkan ke Traefik untuk menambahkan custom header X-Robots-Tag dengan nilai noindex, nofollow ke semua response dari route/path di mana middleware ini dipasang.

Jika sudah di redeploy, kita bisa cek header response-nya dengan curl:

curl -I https://internal-chat.com

Setup robots.txt dengan Nginx#

Ada banyak opsi, yaitu membuat persiten volume dan untuk menempatkan robots.txt atau membuat robots.txt saat compose-nya di deploy (on the fly). Saya memilih opsi kedua, agar tidak perlu membuat volume dan filenya ketika kita pindah/migrasi platform pengelolaan infra.

Kita akan menyisipkan service Nginx minimalis di file docker-compose.yml yang sudah ada. Service ini bakal dikonfigurasi dengan prioritas Traefik yang paling tinggi agar dapat men-serve request /robots.txt sebelum route/path lain pada aplikasi utama.

Ini contoh snippet buat docker-compose.yml kamu:

services:
  app-rails:
  ...
  robots-txt:
    image: nginx:alpine
    restart: always
    command: |
      /bin/sh -c "echo 'User-agent: *' > /usr/share/nginx/html/robots.txt && echo 'Disallow: /' >> /usr/share/nginx/html/robots.txt && nginx -g 'daemon off;'"
    networks:
      ...
    labels:
      - traefik.enable=true
      - traefik.http.routers.robots-txt.rule=Host(`internal-chat.com`) && Path(`/robots.txt`)
      - traefik.http.routers.robots-txt.priority=9999
      - traefik.http.routers.robots-txt.entrypoints=websecure
      - traefik.http.services.robots-txt.loadbalancer.server.port=80
  ...

Service robots-txt ini menggunakan image Nginx versi Alpine yang ringan. Perintah command di atas membuat file robots.txt secara dinamis saat container dijalankan, dengan isi:

User-agent: *
Disallow: /

Artinya, semua user agent (mesin pencari) dilarang mengakses seluruh halaman pada website ini.

Label-label Traefik pada service ini mengatur routing sehingga permintaan ke https://internal-chat.com/robots.txt akan diarahkan ke service Nginx ini. Prioritasnya diatur sangat tinggi (9999).

Redeploy docker compose-nya, lalu cek validasi dengan mengakses https://internal-chat.com/robots.txt pada web browser. Harusnya muncul isi robots.txt yang sudah kita buat.
